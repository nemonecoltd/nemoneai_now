# 2026-07-19 보안 사고 대응 기록

## 발견 경위
"성수/홍대 클릭해도 리스트가 안 바뀐다"는 사용자 제보를 조사하던 중, 원인이 UI 버그가 아니라 **`/ask` 엔드포인트에 대한 SQL 인젝션 스캐너 공격**이 진행 중이었고, 그 부작용으로 서버 전체 응답이 10~20초 이상 느려져 있었던 것으로 밝혀짐. 조사를 확장해 `/feedbacks` 게시판, 인프라 포트 노출, 별도 VM(msm)까지 함께 점검함.

## 공격 현황

### 1. `/feedbacks`(피드백 게시판) 스팸
- 총 181건 중 179건이 SQL 인젝션 스캐너(sqlmap류) 페이로드 또는 빈 값 — `PG_SLEEP()`, `DBMS_PIPE.RECEIVE_MESSAGE()`, `waitfor delay` 등
- 공격 시점: 2026-07-18 12:14경부터 2026-07-19 08:19경까지 총 4차례 버스트
- **실제 인젝션은 실패** — 모든 쿼리가 SQLAlchemy 바인드 파라미터(`:content` 등)로 작성돼 있어 페이로드가 SQL로 실행되지 않고 텍스트 그대로 저장됨
- 근본 원인: `POST /feedbacks`에 서버 측 인증 체크가 전혀 없었음(프론트 UI만 로그인 여부로 폼을 가림) — 봇이 로그인 없이 API를 직접 두드릴 수 있었음
- 부가 발견: 관리자 삭제 기능도 버그였음 — 백엔드가 `user_id == 'admin_uuid_placeholder'`(매직 문자열)로 권한을 체크하는데 프론트는 관리자의 실제 UUID를 보내서, 관리자가 남의 글(스팸)을 삭제할 수 없었음

### 2. `/ask`, `/search`, `/itinerary` — SQL 인젝션 시도 + 실질적 DoS
- `lang`/`region`/`category` 파라미터에 동일한 SQLi 스캐너 페이로드가 계속 유입됨 (2026-07-18 20:55경부터 2026-07-19 조사 시점까지 **진행 중**이었음)
- **실제 SQL 인젝션은 성공하지 않음** — `_lang_col()`이 `lang`이 정확히 "en"/"zh"가 아니면 안전한 기본값(`base`)으로 폴백하는 구조라 임의 문자열이 SQL에 섞여 들어갈 수 없었고, `/places`의 `category` 분기도 알려진 값과 정확히 일치할 때만 f-string에 넣는 구조라 마찬가지로 안전했음
- **그러나 실질적 피해는 있었음**: `/ask`, `/search`, `/itinerary`가 Gemini API를 호출하는 `get_embedding`/`generate_answer`/`generate_walking_tour`가 **동기(블로킹) 함수인데 비동기 핸들러 안에서 그대로 호출**되고 있어서, 공격 요청 하나가 처리되는 동안 uvicorn 이벤트 루프 전체가 멈춰버림. 봇이 반복적으로 요청을 쏘면서 완전히 무관한 `/places` 요청까지 전부 10~20초씩 걸리는 사이트 전역 슬로우다운이 발생 — 이것이 "성수/홍대 클릭이 안 먹힌다"로 체감된 원인

### 3. 인프라 노출 — 포트 직접 노출 (근본 취약점)
- now_back(8081), now_front(3000/3002), matmatch 백엔드(8080), matmatch 프론트(3000)가 전부 `0.0.0.0`에 바인딩되어 있고 **서버에 방화벽 자체가 없어** 공인 IP로 포트만 붙이면 nginx/도메인/HTTPS를 전부 건너뛰고 직접 접근 가능한 상태였음
- 직접 확인: `curl http://34.64.98.113:8081/...` 등 4개 포트 전부 응답함

### 4. msm(주식 서비스) VM 점검
- **현재는 공격받지 않음** — 응답속도 정상(0.35초), pm2 무재시작 13일 유지
- VM 자체 iptables는 비어있지만, **GCP 클라우드 레벨 방화벽이 이미 포트(8080/3000) 직접 접근을 차단** 중 — now/matmatch와 달리 애초에 안전했음
- 과거 이력: 2026-06-25 09:40~09:52(12분간) 동일한 SQLi 스캐너가 `/api/daily`, `/api/stocks/{ticker}` 등을 시도했으나 전부 FastAPI 타입 검증에서 404/422로 거부됨(DB 도달 전 차단). 이후 재발 없음

## 조치 내역

### `/feedbacks`
- 가비지 데이터 179건 삭제(정상 피드백 2건만 보존)
- 인증 구조 전면 개편: `Authorization: Bearer <Supabase access_token>`을 서버가 직접 Supabase `/auth/v1/user`에 검증하는 방식으로 전환 — 클라이언트가 보내는 `user_id`/`admin_email`을 신뢰하던 기존 방식(스푸핑 가능) 제거
- 로그인 필수화(조회/작성 모두), 본인 또는 관리자만 수정·삭제 가능하도록 서버에서 재검증
- 인라인 수정(edit) 기능 신규 추가

### `/ask`, `/search`, `/itinerary`
- `lang` 파라미터를 ko/en/zh 화이트리스트로 검증, 그 외 값은 즉시 'ko'로 대체
- 동기 Gemini 호출(`get_embedding`, `generate_answer`, `generate_walking_tour`)을 `asyncio.to_thread`로 백그라운드 스레드 오프로딩 — 느린 요청 하나가 서버 전체를 막지 못하도록 수정
- 배포 직후 확인: 공격이 계속되는 상태에서도 `/places` 응답 0.07~0.1초로 정상 복구 확인

### 인프라(now + matmatch 서버)
- iptables로 4개 포트(3000, 3002, 8080, 8081)를 `127.0.0.1`만 허용하도록 차단 — nginx는 이미 로컬 경유라 정상 서비스 영향 없음, 도메인 경유 접속(now.nemoneai.com, nemoneai.com) 전부 200 확인
- `iptables-persistent` 설치(약 16KB) 후 규칙을 `/etc/iptables/rules.v4`에 저장 — 서버 재부팅 후에도 유지되도록 처리

### msm VM
- 조사만 진행, 별도 조치는 보류(현재 안전 확인됨, VM 레벨 이중 방화벽 적용 여부는 사용자 결정 대기 중)

## 잔여 과제 / 권고사항
- `create_place`/`update_place`(어드민 전용, `now_back/main.py`)에도 `get_embedding` 동기 호출이 비동기 핸들러 안에 직접 있음 — 관리자만 쓰는 기능이라 우선순위는 낮지만 동일한 블로킹 패턴이라 개선 여지 있음
- 공격자 IP 자체를 차단한 것은 아니라서 스캐너 트래픽은 계속 들어올 수 있음(단, 이제 사이트 성능에는 영향 없음)
- msm VM도 원하면 now/matmatch와 동일하게 iptables 이중 방화벽 적용 가능 — 아직 미결정
